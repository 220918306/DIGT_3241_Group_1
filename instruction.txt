SECTION 0: BEFORE YOU START — UNDERSTANDING YOUR ROLE
================================================================================

What is your job in plain English?
-----------------------------------
You are responsible for making sure three MySQL databases are running inside
the Kubernetes cluster on AWS. Three different backend services (customers,
vets, visits) each need their own database to store data.

Think of it this way:
- customers-service stores owner profiles and pet records → needs customers-db
- vets-service stores vet names and specialties → needs vets-db
- visits-service stores appointment records → needs visits-db

Each database is completely separate. This is the microservices way — no
service touches another service's database.

You do NOT write any application code. You deploy and manage the databases.

What does "deploy a database to Kubernetes" mean?
--------------------------------------------------
Instead of installing MySQL on a physical server, you install it as a container
inside the Kubernetes cluster. The cluster manages it just like any other pod.

You will use Helm to do this. Helm has a pre-built MySQL chart (from Bitnami)
that takes care of all the complicated setup. You just give it a configuration
file (values file) with your database name, username, and password.

What files will you produce?
-----------------------------
1. k8s/mysql-customers-values.yaml  — configuration for the customers database
2. k8s/mysql-vets-values.yaml       — configuration for the vets database
3. k8s/mysql-visits-values.yaml     — configuration for the visits database

You will also create a Kubernetes Secret (via a command, not a file) that
stores the database password securely.

STEP 1.4: Locate and Study the Database Schema Files
------------------------------------------------------
EXPLANATION: "Schema" means the structure of a database — what tables exist,
what columns they have. The schema.sql files tell MySQL what to create.
The data.sql files insert sample data (like fake owners and pets).

  Look at the schema files:

  File 1 — Customers database schema:
    Open: spring-petclinic-customers-service/src/main/resources/db/mysql/schema.sql

    What it creates:
    - types table: pet type (cat, dog, lizard, etc.)
    - owners table: owner name, address, city, phone
    - pets table: pet name, birth date, links to owner + type

  File 2 — Vets database schema:
    Open: spring-petclinic-vets-service/src/main/resources/db/mysql/schema.sql

    What it creates:
    - vets table: vet first name, last name
    - specialties table: specialty names (radiology, surgery, dentistry)
    - vet_specialties table: links which vet has which specialty

  File 3 — Visits database schema:
    Open: spring-petclinic-visits-service/src/main/resources/db/mysql/schema.sql

    What it creates:
    - visits table: pet_id, visit date, description

  Also look at the data files (sample seed data):
    spring-petclinic-customers-service/src/main/resources/db/mysql/data.sql
    (10 owners, 13 pets loaded as sample data)

    spring-petclinic-vets-service/src/main/resources/db/mysql/data.sql
    (6 vets, 3 specialties loaded as sample data)

  IMPORTANT: You do NOT run these files manually. When the customers-service,
  vets-service, and visits-service start up with the mysql profile, Spring Boot
  automatically runs schema.sql and data.sql for you. Your job is only to make
  the empty MySQL database available for them to write to.

STEP 2.2: Create the MySQL Values File for customers-db
---------------------------------------------------------
EXPLANATION: A Helm values file is a YAML configuration file that tells the
Helm MySQL chart how to set up the database: what the database name is, what
the username and password are, and how much disk space to give it.

  Create the file: k8s/mysql-customers-values.yaml

  Copy this content exactly (watch your indentation — YAML is space-sensitive):

  ------------------------------------------------
  auth:
    rootPassword: petclinic-root
    database: petclinic
    username: petclinic
    password: petclinic
  primary:
    persistence:
      size: 1Gi
  ------------------------------------------------

  What each line means:
  - rootPassword: the admin/root password for MySQL
  - database: the name of the database to create (petclinic)
  - username: the regular user account (petclinic)
  - password: password for the regular user
  - persistence.size: 1Gi means 1 gigabyte of disk space

STEP 3.1: Connect kubectl to the EKS Cluster

EXPLANATION: By default, kubectl talks to whatever cluster it was last

    kubectl get nodes

  Expected output: You should see 2-3 nodes in "Ready" status.
  If you see nodes, your kubectl is connected to the right cluster.

STEP 3.2: Check that the Namespace Exists
-------------------------------------------
EXPLANATION: A Kubernetes namespace is like a folder that groups all the
project's resources together. P2 should have created the spring-petclinic
namespace. All your work goes in this namespace.

  Command:
    kubectl get namespaces

  You should see spring-petclinic in the list.
  If it is NOT there, create it:
    kubectl create namespace spring-petclinic

STEP 3.3: Create the Database Secret
--------------------------------------
EXPLANATION: A Kubernetes Secret stores sensitive data (passwords) in a way
that is more secure than putting passwords directly in config files. The
backend services (customers-service, vets-service, visits-service) will
read the database password from this secret, not from a hardcoded config.

  Command:
    kubectl create secret generic petclinic-db-secret \
      --from-literal=mysql-root-password=petclinic-root \
      --from-literal=mysql-password=petclinic \
      --from-literal=mysql-user=petclinic \
      -n spring-petclinic

  Verify the secret was created:
    kubectl get secret petclinic-db-secret -n spring-petclinic

  Expected output: You should see a line showing the secret exists.

  To see what's inside the secret (passwords are base64 encoded, not plain text):
    kubectl describe secret petclinic-db-secret -n spring-petclinic

  NOTE: If you run the create command twice, it will error saying the secret
  already exists. That is fine — just don't run it twice.

STEP 3.4: Deploy the Customers Database
-----------------------------------------
EXPLANATION: This command uses Helm to install MySQL into your Kubernetes
cluster using the Bitnami MySQL chart. The --values flag points to your
configuration file. The name "customers-db" is how the database will be
known inside Kubernetes.

  Command:
    helm install customers-db bitnami/mysql \
      --namespace spring-petclinic \
      --values k8s/mysql-customers-values.yaml

  Expected output: You will see a message from Helm saying the release was
  deployed, plus some instructions (you can ignore the instructions).

  Watch the pod start up:
    kubectl get pods -n spring-petclinic -l app.kubernetes.io/instance=customers-db -w

  Press Ctrl+C to stop watching when the pod shows "Running" status.
  This can take 1-3 minutes.

STEP 3.5: Deploy the Vets Database
------------------------------------

      --namespace spring-petclinic \

STEP 3.6: Deploy the Visits Database

STEP 3.8: Verify the Database Services Exist
----------------------------------------------
EXPLANATION: In Kubernetes, a "Service" is a stable network address for a pod.
Other pods use the Service name (not the pod name) to connect to the database.
For example, customers-service will connect to MySQL using the hostname
"customers-db-mysql" — not an IP address.

  Command:
    kubectl get svc -n spring-petclinic | grep db

  Expected output:
    customers-db-mysql       ClusterIP   10.x.x.x   3306/TCP
    vets-db-mysql            ClusterIP   10.x.x.x   3306/TCP
    visits-db-mysql          ClusterIP   10.x.x.x   3306/TCP

  These internal addresses are how the backend services connect to their
  respective databases.

STEP 3.9: Test the Database Connection (customers-db)
------------------------------------------------------
EXPLANATION: This runs a temporary MySQL client container inside the cluster,
connects to your database, and runs a test query. It verifies the database
is actually accepting connections and the credentials work.

  Command:
    kubectl run mysql-client --rm -it --image=mysql:8 -n spring-petclinic \
      -- mysql -h customers-db-mysql -u petclinic -ppetclinic petclinic \
      -e "SHOW TABLES;"

  Breaking this command down:
  - kubectl run mysql-client: create a temporary pod named mysql-client
  - --rm: delete the pod automatically when done
  - -it: attach to it interactively (so you see the output)
  - --image=mysql:8: use the MySQL 8 container image
  - -n spring-petclinic: put it in the right namespace
  - -- mysql ...: the MySQL client command to run inside the pod
  - -h customers-db-mysql: connect to the customers-db MySQL service
  - -u petclinic: username
  - -ppetclinic: password (note: no space after -p)
  - petclinic: the database name
  - -e "SHOW TABLES;": run this SQL command

  Expected output (after Spring Boot has initialized the DB — see note below):
    +---------------------+
    | Tables_in_petclinic |
    +---------------------+
    | owners              |
    | pets                |
    | types               |
    +---------------------+

  NOTE: The tables only appear AFTER the customers-service has connected to
  the database and run schema.sql. If you test right after deploying MySQL
  (before deploying customers-service), you might see "Empty set" — that is
  normal. The tables get created when the app first starts.

  If you see "ERROR 1045 (28000): Access denied", the username/password is
  wrong. Double-check your secret and values file.

SECTION 8: TROUBLESHOOTING GUIDE
================================================================================

PROBLEM: "helm install" fails with "release already exists"
SOLUTION: The database was already deployed. Check with:
  helm list -n spring-petclinic
  If it's there, don't deploy again. If you need to redeploy:
  helm uninstall customers-db -n spring-petclinic
  Then run the helm install again.

PROBLEM: MySQL pod stays in "Pending" status forever
EXPLANATION: Pending means Kubernetes cannot find a node to schedule the pod on.
Most likely the EKS nodes are full or the PVC cannot be created.
SOLUTION:
  kubectl describe pod <pod-name> -n spring-petclinic
  Look for "Events" at the bottom — they explain why it's pending.
  Tell P2 that you need more node capacity.

PROBLEM: MySQL pod shows "CrashLoopBackOff"
EXPLANATION: The pod starts, crashes, restarts, crashes again in a loop.
SOLUTION:
  kubectl logs <pod-name> -n spring-petclinic --previous
  This shows logs from the previous run (before the crash).
  Common cause: wrong values in your values.yaml file.

PROBLEM: "Access denied for user 'petclinic'"
EXPLANATION: The username/password in the secret doesn't match what was set
in the Helm values file.
SOLUTION: Make sure mysql-password in the secret matches password in values.yaml.
Both should be "petclinic".

PROBLEM: "No route to host" or connection refused
EXPLANATION: The service name being used doesn't match the actual K8s service name.
SOLUTION:
  kubectl get svc -n spring-petclinic | grep mysql
  The EXACT names shown there are what the backend services must use as hostname.
  customers-db-mysql, vets-db-mysql, visits-db-mysql

PROBLEM: kubectl says "Unable to connect to the server"
EXPLANATION: kubectl is not configured to talk to the EKS cluster.
SOLUTION:
  aws eks update-kubeconfig --name petclinic-cluster --region us-east-1
  Then try again.

PROBLEM: "SHOW TABLES" returns empty set after customer-service is running
EXPLANATION: The service uses HSQLDB (in-memory) by default, not MySQL.
The mysql Spring profile must be active for it to use MySQL.
SOLUTION: Check with P5/P6 that their deployment manifest includes:
  - name: SPRING_PROFILES_ACTIVE
    value: "docker,mysql"
  Without the mysql profile, the service uses an in-memory database and
  your MySQL database will remain empty.

  helm uninstall customers-db -n spring-petclinic

Redeploy after uninstall:
  helm install customers-db bitnami/mysql \
    --namespace spring-petclinic \
    --values k8s/mysql-customers-values.yaml

Connect to a running MySQL pod directly (interactive):
  kubectl exec -it <mysql-pod-name> -n spring-petclinic -- mysql -u petclinic -ppetclinic petclinic

Get all events (useful for debugging):
  kubectl get events -n spring-petclinic --sort-by='.lastTimestamp'
